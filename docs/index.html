Works - Corporate Site

コーディングに関わったコーポレートサイトのうち、2017年現在の直近のものをご紹介いたします。
基本的にはJavaScriptのコーディングが主な担当であり、関連するHTMLとCSSについてもコンポーネント単位で作成しました。 プロジェクトの方針次第では、JavaScript側からHTML,CSSの生成を行い、メンテナンスの容易さを優先した構成となることを優先しています。

大手複合商業施設サイト

担当業務
  • JavaScriptコーディング(全範囲)
  • 一部のCSS, HTMLコンポーネント作成
(引き継ぎ済であり、その後に改修の入っている箇所もあります)
コーディング形態
  • jQuery, vue.js 使用
  • アニメーションや特殊UIはスクラッチでの開発
  • 開発期間40日程度＋公開後に断続的な改修

特殊なUIを多く含むものの、開発期間が限られてライブラリを選択する時間的な余裕も難しかったこともあり、 jQueryは要素選択やCSS操作程度に使用、JSテンプレートエンジンはvue.jsを使用、それ以外については ほぼスクラッチでの開発となりました。

開発当初のトップページは HTML記述の可能な巨大なカルーセルを使用し、ナビゲーションをその上に被せて表示するものだったため、非常に難易度の高いコーディング技術が要求されました。その際、ロゴのSVGが背景色に溶け込むなどの影響もあり、canvasで動的に平均色を割り出した色範囲に対応してロゴを入れ替えるなどの処理も加えました。 リリース直前に仕様が変わって余白の多いデザインに代わりましたが、アニメーションタイプや各サイズの設定値をまとめた汎用的なつくりにしていたため、頻繁な仕様変更に対応可能な構成になっています。

特殊UIとして特筆すべきものに「フロアマップ」が挙げられます。
GoogleMapsと同様の操作感を実現した画像ビューアが必要とされ、アニメーション付きズーム（ピンチイン・ピンチアウト）や慣性を効かせたスクロール、特定の座標へのピン表示など、すべてスクラッチで作成しました。（座標変換や指数対数計算などが絡み、数学的にも難しいものでした）

都内私立大学 入試サイト

担当業務
  • JavaScriptコーディング(リニューアル時の改修)
  • その他 特設サイトのJSコーディング

基幹システムのログイン情報に関するAPIと連携したJavaScriptのメンテナンスを担当しました。 主に、以前 Backbone.js + CoffeeScript + jQuery を使用して作成されたコードに潜伏している不具合の発見・改修を行い、単体試験については専用のブックマークレットを作成して対応しました。

その他

サイトに使用するコンポーネントのうち、JavaScriptへの依存性の高いものについては、メイン担当業務以外のサイトについても担当いたしました。 多くは、

  • グローバルナビ（メガメニュー）とスマホ版のアコーディオンメニューの連携
  • TabUIとstickey要素の管理
  • カルーセル（自動送り・手動送り双方）

等を扱い、いずれも一般的なJSライブラリでのカスタマイズの範囲を超えた特殊なデザインのものを担当しております。

特殊効果カルーセル

車のワイパーのような動きで画像を切り替えたい という要件があり、canvas要素で対応しました。 ２枚のcanvasをレイヤーとして重ね合わせることで動作の軽いカルーセルとして仕上げています。
カルーセル内に検索フォーム、さらに下部中央にも円形のアクセサリが配置され、これらの大きさと位置は画面サイズに完全に依存するため、 (css3で新しく導入された)vwを単位として使用し、極力JavaScriptを使用せずにcssで済むように工夫しています。

クリッカブルマップ

日本地図からエリアを選択したい という要件があったものの、HTMLでの <map><area/></map> はデータの作成とメンテナンス性に難があるため、
一定値以上のalphaを超える同色で塗られた領域を１つのエリアとみなすJSプログラムを作成し、PNG画像から自動的にクリッカブルマップとして認識させています。この画像の解析にもcanvasを使用し、ピクセルデータの扱いで動作が重くならないよう、アルゴリズムにも工夫しました。

Works - BookOS UI β

コンテストに出展した OS風サイトでの小説閲覧サイト「山猫文庫」をご紹介いたします。
縦書きをテーマとしたコンテストではありましたが、webアプリケーションと縦書きの親和性を確認したいと考え、UIとして「直観的な操作性をもったプラットホーム」と「読書環境としてなじみのあるインターフェイス」を構築しました。

山猫文庫

URL
http://ihatov.moon-bow.net/
担当範囲
コンテンツ用画像を除く全域（コンテンツ画像は妻が担当）
コーディング形態
  • コーディング期間５日程度（コードは非常に雑です）
  • Back-endではWordPressを使用。themeはフルスクラッチで作成。
  • Front-endではJavaScriptをフルスクラッチで記述。
仕様面の特徴
  • 小説の表示については、ページめくりをできる本のようなインターフェイスを作成し、現実の読書の感覚と近くなるようにしています。(ページめくりのアニメーションは、このサイトで使用している3D回転とほぼ同じものです)
  • 将来的にユーザー同士のコミュニケーションも可能な統合環境を前提とし、扱いやすいOS風のインターフェイスを作成しました。
  • 出展の要件として「CSSでの縦書き」が挙がっていたため、縦書き表示を前提とした「本」を実装しています。
  • 電子書籍のように１ページ毎で閉じた場合、好みのフォントサイズやページサイズを利用できないため、文章コンテンツはリフローさせて動的にページを生成しています。

縦書きデザインのコンテストへの出展のため、CSSの縦書きモジュールを使用することが必須となり、加えて、リフロー処理にCSSのcolumnsを使用しているため、表示処理速度は非常に遅いものとなってしまいました。 そのため、電子書籍の形態としては β版として考えています。
ブラウザそれぞれで、縦書き指定の writing-mode:vertical-rl と columns を同時に使用することがまだ前提になっていないようであり、このサイトを作成した後に（ブラウザのバージョンアップで）発生した表示不具合もいくつか見られます。 次のリファクタリング時には columnsの使用をやめ、リフロー処理用に新規でJSコードを用意することで表示速度の向上を図る予定です。

Works - JP Promotion

コンテストに出展した 絵巻物風インターフェイスの日本文化紹介サイトです。
与えられたコンテンツをそのまま使用することが条件であるため、レイアウトの統一感などが欠けてしまっていますが、（特に日本語が分からない方にとって）分かりづらいコンテンツ内容をUIでカバーするよう心がけました。

日本文化事始め －絵巻物で体感する日本文化－

URL
http://jpculture.moon-bow.net/
担当範囲
コーディング全範囲（テキスト・画像は事務局からの提示コンテンツ）
コーディング形態
  • コーディング期間７日程度（コードは非常に雑です）
  • 静的HTMLとして作成。コンテンツは、素のテキストにセクショニング要素等を追加。
  • UIに関してJavaScriptを使用。フルスクラッチで記述。
仕様面の特徴
  • 日本文化をテーマとしているため、日本語が読めなくても雰囲気である程度の内容が伝わるよう、絵巻物風のインターフェイスを作成して使用しています。
  • 序章と各論３章を合わせた４つのセクションに対して、それぞれを１つのパネルとし、横スクロールのアコーディオンを使用して表示しています。
  • 日本文化のイメージを具体化するため、UIにリアリティを持たせるよう画像素材をCSSに併用, 裏地に透けた文字をCSSで表すなど工夫をしています。
  • 時系列に関連するコンテンツについては、パネル内でパララックス効果を持たせ、グラデュアルな時間変化を表現しています。
  • （要件には挙がっていませんでしたが）レスポンシブ対応済みです。

コンテストの「課題部門」だったためコンテンツの縛りがあり、レイアウトに苦戦しました。 また、縦書きコンテンツを含む場合のレスポンシブ対応は ブラウザ側の表示仕様の差異の影響を大きく受け（columns使用時のheightに関する解釈の差が大きい）、ブレイクポイントに画面の高さも考慮しているため、コンテンツの表示調整に一番工数が掛かりました。
コンテストの協賛業者や審査員のリストが、出版等タイポグラフィに拘りの強い傾向があったため、CSSのtransform:rotateを使用して MS明朝のようなジャギーの発生するフォントでも見た目に耐えられる表示にしました。

Works - JS Captcha

個人事業として開発し汎用的に使用している小規模SNS向けの wordpress用 会員管理プラグインです。セキュリティ強化のためJavaScriptによる画像認証を併用しています。
wordpress用の会員管理プラグインは多く存在するものの、メール配信内容やログイン画面など各画面のカスタマイズができないものが多く、部分的に英語が残るなどしてしまうため、オリジナルのプラグインとして作成しました。（今のところ、まだ配布サイトには登録していません）

URL
会員登録・ログイン機能用サンプルサイト http://dev.moon-bow.net/
（画像認証のパターンは「魚群」「どんぐり」「桜」「もみじ」「雪景色」の５パターンを用意していますが、これらはwordpress管理画面で サイト管理者が画像や動作設定を行うことになります）
担当範囲
全範囲
プラグインの特徴
  • インストール後に専用のカスタム投稿タイプページが自動的に作成され、login画面などではこの専用ページを表示します。
  • 通常のlogin画面は使用不可となるため、スパム攻撃をかわしやすい構成になっています。ログイン処理自体はwordpressのデフォルトの関数を使用しているため、セキュリティ面での劣化はありません。
  • 専用ページ表示用のtemplateはデフォルトで用意してあり、同名のtemplateを使用しているtheme内に配置することで上書くことができます。（single.phpなどと同じ感覚でカスタマイズできます）
  • 会員登録時の処理フローは自動で行われ、確認メール配信やその後のアクティベーションも機能に含まれます。
  • 配信する確認メールは、テンプレートをwordpress管理画面にて編集することが可能です。
  • 会員として登録した場合、自動的に編集者権限がふられます。管理者権限以外のユーザーはwordpress管理画面には入れず、login情報のみが保持されます。（single.php や page.php 内でのlogin情報による処理分岐を想定）
画像認証の特徴
  • 背景画像とドット画像（複数パターン可）の登録が可能で、必要に応じてアニメーションパターンを設定することができます。
  • 登録された画像認証パターンは、ランダム表示 や 月ごとの切り替えが可能です。
  • 設定済みパラメータを使用して、動的に画像認証用JavaScriptコードを出力します。
  • ドットが形成する認証文字列（数字）はAjaxで取得した座標データを展開して表示します。座標データは実際の数字のpathからランダムに数ピクセルずらしたものを使用し、その順番もシャッフルしたデータをresponseとして返しているため、悪意のある自動解析プログラムでの解析が困難なものとなっています。

Front-end Technical Remarks

フロントエンド開発における、私自身のコーディングパターンやポリシーについて、概要を記載いたします。
基本的には（下請けや個人事業などの）個人開発を前提としたポリシーですので、チーム開発の場合はそのプロジェクトのポリシーを遵守します。

フロントエンド開発 全般について

  • 新規構築の場合、メンテナンス人員とサイトの寿命を意識します。期間限定のサイトで長い寿命を必要としない場合はパフォーマンスを重視し、場合によってはハードコーディングすることもあります。 メンテナンスを前提とした長寿命サイトの場合は、メンテナンス人員の技術レベルに合わせて構成を検討します。（無理にMVCモデルに当てはめることは避けるようにしています）
  • 上記「サイト寿命」の問題に関連し、流行りのフレームワークの使用にはやや消極的です。
    ただし、Ajaxで取得したJSONの展開に関しては例外的であり、vue.jsなどをJSテンプレートエンジンとして好んで使用します。
  • node.js等を使用したタスクランナーについても、メンテナンス人員の技術レベルに合わせて検討します。

HTMLについて

  • 案件ごとの対応ブラウザを確認し、プロジェクトで定められたHTMLバージョンに対応したマークアップを行います。
  • なるべく文脈に合わせた要素選定をしていますが、サーバー側のテンプレート化の可能性 や 別の作業者による「コピペ」使用の場合も考慮し、汎用的な要素を使用することも多々あります。
  • デザイン上やむを得ず必要な要素については、CSSの疑似要素を優先的に使用し、それでも補いきれない場合はJavaScriptで補完します。
  • SEOもユーザビリティも意識する必要のないサイトについては、メンテナンス性を優先したマークアップを行います。

CSSについて

  • BEM記法など、セレクタの命名規則については、プロジェクト方針に依存します。
  • HTML自体が(UL-LIなど)親子縛りのある要素である場合、クラス名の乱用は避け、親のクラス名と子セレクタを使用するようにしています。
  • JavaScriptで要素を取得するためのクラス名は使用せず(例：js-xxx)、data属性を使用する方です。
  • margin や padding の付与規則はあまり厳密に設定しない方です。（多くの場合、デザインポリシーがしっかり固まっていれば、これらの指定に困ることは少ない傾向にあります）
  • z-indexの付与規則は比較的こだわる方です。状況次第では、JavaScriptにdebug関数を埋め込みながら開発を進めます。
  • アニメーションは（JavaScriptのタイマーを使用するのではなく）CSSのtransitionを優先して使用します。transitionの設定値が動的な場合でも、transition-functionが特殊でない限りはCSSを使用したアニメーションを利用します。

JavaScript(ブラウザJS)について

  • altJSは、現状は必要性を感じられないため使用していません。型を気にするためにTypeScriptを使用したいならば、型を気にしながらJavaScriptでコーディングすれば済み、オブジェクト指向を意識したコードを書きたければ関数をnewすれば済むためです。
  • CoffeeScriptについても個人では一切使用しません。変数のスコープが１つグローバルなスコープで宣言されるため、バグが潜伏している記述を多く見かけます。
  • エラーを隠ぺいする記述は避け、積極的に例外をthrowします。これはjQuery使用時にバグを発生させやすく（特にlength==0の対応）、深刻な不具合を早期に見つけるための対応です。
  • 関数オブジェクトのmethodやprototypeの操作は、ライブラリ作成時以外では使用しません。
  • thisキーワードについては、基本的には、関数内でpublicな値の指定を行う場合以外には使用しません。jQueryなど、thisキーワードを強要するライブラリは多く存在しますが、その場合もすぐに別の具体的な意味をもつ名前の変数に置き換えて使用します。

  • SVNの使用経験はありますが、10年近く触っていないため、操作にやや不安があります。
  • Gitの基本操作は問題なくできている（はず）です。コマンドも使用しますが、操作ミスを防ぐためにもなるべくGitクライアントを使用しています。
  • Gitフローの管理も可能です。作業者としては、むやみにマージして逆流フローを発生させず、branchのtreeの可読性を維持するようにしています。(PullRequestを必須としたプロジェクトは除く)

エディタについて

  • IDEを指定されたプロジェクトの場合、その指定環境を使用します。
  • 指定が無い場合、windowsサクラエディタを使用します。こだわりはありません。
  • (TitaniumStudioで全ファイルを消失 → 完全復旧不能 となったことがあり、IDEに多少の恐怖心はあります。)

OSについて

  • 開発環境としては、ここ４年近くはwindowsを扱っています。IEでのデバッグが主な目的ですので、それ以外のこだわりはありません。
  • 動作確認用に、個人所有の端末として macOS(PC), iOS(iPhone5s, iPad mini), android(dtab2, Xperia, nexus7)を所有しています。

Back-end Technical Remarks

バックエンド開発における、私自身のコーディングパターンやポリシーについて、概要を記載いたします。
基本的には（下請けや個人事業などの）個人開発を前提としたポリシーですので、チーム開発の場合はそのプロジェクトのポリシーを遵守します。

バックエンド開発 概要

バックエンドに関しては、現在は基本的に WordPress のみを扱っております。必然的にPHPについても仕様全般をおさえていますが、フレームワークやツールに関しては手を付けていない状況です。
10年前まではJava(J2EE)を本職で使用していましたが、現在は辛うじて概念を記憶している程度です。
DBについても、リレーショナルデータベースに関してsql程度（MySQL）は扱えますが、チューニング等は完全に守備範囲外になります。

WordPressについて

  • バックエンドの中でも、WordPressに関しては極端に守備範囲が広く、プラグインやテーマの作成も可能です。
  • 配布を目的としていない場合にも、プラグインは頻繁に作成します。汎用的に扱うことが望ましい機能についてはプラグイン化を、サイト固有の機能についてはテーマにてハードコーディングをも含む記述を行っています。
  • configなどサイトで持つべき設定値は、極力、option値としてサイト側に持たせて管理します。その際、管理画面から入力可能なUIを、テーマのfunctions.phpで作成します。
  • カスタム投稿タイプやカスタムフィールドを活用したwebアプリケーション化も得意とし、これに合わせた管理画面のカスタマイズも不自由なく行えます。また、管理画面以外のページでもログイン状態に合わせた表示の切り替えを行うことも可能です。
